# ADR-007: 管理者監査ログ

## ステータス

承認済み

## 日付

2026-03-01

## コンテキスト

本番環境の管理者Webにおいて、以下の課題がある:

1. **不正操作の検知**: 管理者の書き込み操作（作成・更新・削除）を記録する仕組みがなく、不正操作の発見が困難
2. **障害時の操作追跡**: 不具合発生時に、どの管理者がいつ何を操作したかを追跡できない
3. **コンプライアンス**: OWASP 2025 / NIST SP 800-92 では、特権操作の監査ログ記録がベストプラクティスとして推奨されている

## 決定

Ginミドルウェア方式による監査ログを実装する。

### 方式
- **記録方式**: Ginミドルウェアで全POST/PUT/PATCH/DELETEリクエスト + 認証イベントを自動記録
- **書き込み方式**: 非同期バッチ書き込み（バッファードチャネル + バッチINSERT）
- **保存先**: TiDB `users.audit_logs` テーブル（追記専用）
- **PII方針**: リクエスト/レスポンスボディは保存しない。変更フィールド名のみ記録

### 記録対象
- POST / PUT / PATCH / DELETE リクエスト（認証済み）
- SignIn 成功/失敗、SignOut

### 記録対象外
- GET リクエスト（読み取り操作）
- ヘルスチェック
- Webhook
- トークンリフレッシュ

## 検討した選択肢

### 選択肢1: Ginミドルウェア方式 (採用)

- 利点:
  - 全エンドポイントを自動的にカバーでき、記録漏れが起きにくい
  - 既存のハンドラコードへの変更が最小限
  - HTTPレイヤーの情報（ステータスコード、レイテンシ等）を自然に取得可能
- 欠点:
  - ビジネスロジック内部の詳細な変更内容は取得しにくい
  - ルートマッピングの保守が必要

### 選択肢2: イベントソーシング方式 (却下)

- 利点:
  - ビジネスイベントレベルの詳細な記録が可能
  - リプレイ可能な変更履歴
- 欠点:
  - 全サービスメソッドの変更が必要で、実装コストが非常に高い
  - 既存アーキテクチャとの整合性が低い

### 選択肢3: DBトリガー方式 (却下)

- 利点:
  - アプリケーションコードの変更が不要
  - 記録漏れが原理的に発生しない
- 欠点:
  - 操作者（adminID）やHTTPコンテキストを取得できない
  - TiDBのトリガーサポートに制限がある
  - パフォーマンスへの影響が大きい

## 結果

### 期待される効果

- 管理者の全書き込み操作が自動的に記録され、不正操作の検知・追跡が可能になる
- 障害調査時に操作履歴を参照でき、原因特定が迅速になる
- 非同期バッチ書き込みにより、リクエストレイテンシへの影響を最小化

### 受け入れるトレードオフ

- 非同期書き込みのため、チャネル満杯時はログがドロップされる可能性がある（リクエスト処理優先）
- リクエスト/レスポンスボディを保存しないため、変更された値の詳細は追跡できない
- ルートマッピングテーブルの保守が必要（新規エンドポイント追加時に更新が必要）
